# SOLID Analyzer

> **パス基準**: このファイル内の全てのパスは **このスキルディレクトリからの相対パス**（`references/xxx.md`）です。評価対象のソースコードはプロジェクトルート基準の絶対/相対パスで扱います。

指定された **単一コンポーネント** のSOLID原則（SRP/OCP/LSP/ISP/DIP）適合性を、その粒度（Gradleモジュール/パッケージ/クラス/関数）に応じた基準で評価する。原則の定義・出典は `references/solid-principles.md`、粒度×原則の翻訳表と判定チェック項目は `references/granularity-criteria.md`、レポート構成は `references/solid-output-format.md` を正とする。

## 前提

- 対象は **Kotlin/Androidプロジェクト**（マルチモジュール、Dagger Hilt、Coroutines等を想定）。
- このスキルは **評価と処方の提示まで** を行い、リファクタリングの実装はしない。
- 網羅的なモジュール間結合分析や差分レビューは別スキルの担当（`description` の Do NOT use for 参照）。

## 引数

自由形式の文字列1つ。モジュールパス（`:domain`）、パッケージ（`tv.abema.domain.user`）、クラス名（`UserRepositoryImpl`）、関数名など、粒度の指定は問わない。

1. Glob/Grep で実体を解決し、粒度を **自動判定** する（判定基準は下記「粒度の判定」）。
2. 解決結果による分岐:
   - **1件に確定** → 確認せず進める。
   - **0件** → 検索条件を変えて再試行（大文字小文字無視、部分一致、パッケージ名の別解釈）。それでも0件なら **推測で進めず**、見つからなかった旨と試した検索条件をユーザーに報告して終了。
   - **複数マッチ**（同一粒度で複数の実体、または複数の粒度にまたがる: 例 クラス名がパッケージ名にも一致）→ `AskUserQuestion` で候補を提示して確認。

### 粒度の判定

表を **上から順に** 評価し、最初に該当した粒度を採用する（形式的な手掛かりが強い順）:

| 手掛かり | 粒度 |
|---|---|
| `:xxx` 形式、または `build.gradle.kts` を持つディレクトリにマッチ | Gradleモジュール |
| パッケージFQN、または複数の `.kt` を含むソースディレクトリ | パッケージ |
| 単一の class/interface/object 宣言 | クラス |
| 単一の fun 宣言 | 関数 |

形式的な手掛かりがなく（裸の名前など）、複数の粒度の実体にマッチし得る場合は「複数マッチ」として上記の確認フローに従う。

## ワークフロー（粒度でスケールする）

### クラス / 関数 粒度 → メインエージェントが直接評価

サブエージェントは起動しない。以下を読んで直接評価する:

1. 対象の全文。
2. その **依存先**（import しているクラス・呼び出している関数）と **呼び出し元**（対象を使う側）を Grep で特定して読む。LSP評価では継承階層（親クラス/実装インターフェース、他の実装）も読む。
3. `references/granularity-criteria.md` の該当粒度の行に沿って原則別に judge する。
4. 会話に **レポート全文を直接出力**（`references/solid-output-format.md` 準拠）。ファイル保存はしない。

### パッケージ / モジュール 粒度 → Opusワーカーを並列起動

1. 対象内のクラス群（とモジュールなら `build.gradle.kts` の依存宣言）を **意味的なまとまり** で分割する（例: 機能グループ単位、公開API群/内部実装群など）。
2. 分割した各範囲を `Agent` ツール（`subagent_type: general-purpose`, `model: opus`）に割り当て、**1メッセージ内で複数の Agent を同時に投げて並列起動** する。
3. 各ワーカーへ渡すもの:
   - `references/granularity-criteria.md`（**読ませる**）と `references/solid-principles.md`。
   - 担当範囲のパス一覧、対象の粒度。
   - 指示: 実ソースを読み、担当範囲について **原則別の所見**（verdict候補 + 根拠 `file:line` + 重大度候補 + 実害シナリオ + 処方の方向性）を返させる。投機的抽象化を勧めず、実害シナリオを伴わない指摘は「軽微な逸脱」以下に留めるよう明記する。
4. メインエージェントがワーカー結果を **集約し、最終verdictを確定** する（ワーカー間で食い違う判定はメインが実ソースを確認して裁定）。

**モジュール粒度の主眼**: 「モジュールとしての評価」を主とする — 公開API面（他モジュールから見える表面積）、依存方向（DIPが層をまたいで守られているか）、**変更理由の単一性**（このモジュールが応答するアクターは単一か）。内部クラス個々のSOLID違反は、モジュールの問題を示す **代表例として引く程度** に留め、内部クラスの網羅監査はしない。

## 評価表現

各原則を **4段階verdict** で判定する:

- **遵守** — 原則に適合。
- **軽微な逸脱** — 原則から外れるが、実害シナリオが弱い/限定的。処方は任意。
- **違反** — 具体的な変更シナリオで実害が説明できる。重大度・根拠・処方を必須で付す。
- **評価対象外** — その粒度でその原則が適用不能（例: 関数のISP/LSP）。理由を書く。

**違反** には必ず付す:
- **重大度**: high / medium / low。
- **根拠**: `file:line`（具体的なFQCN/パス）。
- **実害シナリオ**: 「どういう変更が来たときに、なぜ困るか」を具体的に。
- **処方（強度: 方向性まで）**: リファクタの方向性（例「Aインターフェースを画面単位に分割」）を示す。**実装はしない。**

**総合点は出さない。** 違反を重大度順（high→low）に並べる。verdictの4段階と、違反/軽微な逸脱の境界（実害シナリオが説明できるか否か）の判定フローは `references/granularity-criteria.md` に従う。レポートの構成は `references/solid-output-format.md` に従う。

## 出力先

- **クラス / 関数** → 会話に **レポート全文を直接**出力。
- **パッケージ / モジュール** → 会話に **エグゼクティブサマリ**（3行以内）を出力し、**全文を `tmp/docs/solid-<対象名>.md`** に保存。
  - 保存前に `tmp/` が **git管理下でないか** 確認する（`git check-ignore tmp/` 等）。`.gitignore` されていなければ、コミットに含まれ得る旨を **注意喚起** してから保存する。

## キャリブレーション注意

- **原則の過剰適用を戒める。** SOLIDは目的ではなく手段。違反判定は「現に困る、または困る具体的シナリオがある」ことを伴わせる（`references/solid-principles.md` の批判・限界セクション参照）。
- **投機的抽象化を推奨しない。** 「将来拡張するかもしれないから interface を切れ」式の指摘はしない。今ある変更軸・実害に基づく。
- 深いモジュール（単純な公開面 + 複雑な実装）はしばしば良い設計であり、SRP/OCPを理由に表面積を増やす提案はアンチパターンになり得る。保守的に判定する。
