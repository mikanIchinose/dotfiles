# 粒度別 判定基準（粒度 × 原則の翻訳表）

このスキルの心臓部。同じ原則でも粒度によって「何を意味するか」が変わる。まず対象の粒度を確定し、下表の該当行を読む。翻訳の根拠は `solid-principles.md` のコンポーネント原則セクション。

**verdictの4段階**: 遵守 / 軽微な逸脱 / 違反 / 評価対象外。判定フローは末尾「verdict判定フロー」参照。

---

## 翻訳マトリクス（4粒度 × 5原則）

各セル: 「その粒度でその原則が何を意味するか」（1-2文）。

### SRP（単一責任 / 変更理由の単一性）

| 粒度 | 意味 |
|---|---|
| **Gradleモジュール** | モジュールが応答する **アクターが単一** か。翻訳根拠: **CCP**（同じ理由で変わるクラスを集める）+ アクター単一性。複数ドメインの変更要求がこのモジュールに集まるなら違反。 |
| **パッケージ** | パッケージ内クラスが **同じ変更理由で束ねられている** か。関心事の混在（UI整形とビジネスルールが同居等）は逸脱〜違反。 |
| **クラス** | クラスが応答するアクターが1つか。**メソッド数ではなくアクター数** で判断（`solid-principles.md` 参照）。 |
| **関数** | 関数が **単一の抽象レベルの単一の仕事** をしているか。副作用と計算の混在、複数の変更理由を1関数に抱えていないか。 |

### OCP（拡張に開き修正に閉じる）

| 粒度 | 意味 |
|---|---|
| **Gradleモジュール** | 新機能追加時にこのモジュールを **書き換えずに拡張** できるか（プラグイン点/抽象を公開しているか）。翻訳根拠: CCP+OCP。 |
| **パッケージ** | 新種の追加がパッケージ内の既存クラス編集を強いるか、それとも新クラス追加で済むか。 |
| **クラス** | 新しい種類の振る舞い追加時に、既存の分岐（when/if連鎖）を編集して回る構造になっていないか（ポリモーフィズムで開けているか）。 |
| **関数** | 新種の入力型が増えるたびに関数内の分岐を書き換える必要があるか。**戦略/高階関数で外から差せる** か。 |

### LSP（置換可能性 / 契約の保存）

| 粒度 | 意味 |
|---|---|
| **Gradleモジュール** | 原則として **評価対象外**（モジュールは置換される抽象ではない）。例外: 同一 interface を差し替える複数実装モジュール（例 flavor/variant で bind 先を差し替える）がある場合のみ、契約保存を評価。 |
| **パッケージ** | 同上、原則 **評価対象外**。パッケージ内に共通の抽象を実装する複数クラス群があれば、その契約保存を代表例として引く。 |
| **クラス** | 継承階層/インターフェース実装で **事前条件を強化・事後条件を弱化・不変条件を破壊・履歴制約に違反していないか**（`solid-principles.md` の4条件）。 |
| **関数** | 原則 **評価対象外**。例外: 関数型パラメータ（`(T) -> R`）を受け、差し替えられる関数実装が契約（同じ定義域で例外を投げない等）を守るべき文脈があれば評価。 |

### ISP（インターフェース分離 / 使わないものへの依存回避）

| 粒度 | 意味 |
|---|---|
| **Gradleモジュール** | このモジュールが公開する **API面（他モジュールから見える表面積）** が、各利用モジュールにとって過大でないか。翻訳根拠: **CRP**（一緒に使われないものを同梱しない）。 |
| **パッケージ** | パッケージの公開シンボル群が、利用側ごとに「使わないもの」を強制していないか（CRP）。 |
| **クラス** | 公開インターフェースが **クライアントごとに** 太すぎないか。全クライアントが全メソッドを使うか。 |
| **関数** | 原則 **評価対象外**（関数は分離すべき「複数メソッドのインターフェース」を持たない）。ただし巨大な引数群（多すぎるパラメータ、使われない引数の常時 null 渡し）はISP的兆候として軽微な逸脱で言及し得る。 |

### DIP（依存方向を抽象へ）

| 粒度 | 意味 |
|---|---|
| **Gradleモジュール** | **依存方向が層の安定性に沿い、抽象へ向いているか**。翻訳根拠: **SDP+SAP**。上位層モジュールが下位層の **具象** に直接依存していないか（domain が data の実装を掴んでいないか）。 |
| **パッケージ** | パッケージ間依存が具象方向に流れていないか。上位パッケージが下位の実装詳細を import していないか。 |
| **クラス** | クラスが協調相手を **抽象（interface）越しに** 保持しているか、下位層の具象クラスを直接 `import`/`new` していないか。 |
| **関数** | 関数が引数/レシーバとして **抽象** を受けているか、内部で下位層の具象を直接生成・呼び出していないか。 |

**モジュール/パッケージ行の翻訳根拠まとめ**: モジュールのSRP ≒ **CCP + アクター単一性**、モジュールのISP ≒ **CRP**、モジュールのDIP ≒ **SDP + SAP**、モジュールのOCP ≒ **CCP + OCP**。

## Kotlin/Android 固有の判定チェック項目

各セルの判定を、このプロジェクトの具体的な観察対象に落とす。「兆候」であって、単独で違反確定ではない（実害シナリオで裏付ける）。

### SRP
- **モジュール**: `build.gradle.kts` の依存が **複数ドメインに跨る**（例 決済 + 視聴履歴 + 認証を1モジュールが束ねる）。変更コミットの理由が混在する（機能追加とインフラ差し替えが同じモジュールに集中）。
- **クラス**: 1クラスが「API通信 + DB永続化 + UI向け整形」を同時に持つ。ViewModel が画面状態管理に加えて業務ルール判定まで抱える。
- **関数**: 1関数がバリデーション + 変換 + 副作用（保存/送信）を直列で全部やる。

### OCP
- **クラス/関数**: `when(type)` / `if-else` の **型分岐連鎖** が複数箇所に散在し、新種追加のたびに全箇所を編集する。→ 良い形は **`sealed class` + `when` 網羅**（新種追加時にコンパイラが網羅漏れを検出）や、ポリモーフィックな dispatch。
- `sealed` + `when` でも、同じ分岐ロジックが **複数箇所にコピー** されていれば逸脱（1箇所ならOCP的にはむしろ健全なことが多い — 過剰に抽象化しない）。

### LSP
- **継承よりコンポジション** が守られているか。`open class` のオーバーライドで **事前条件を強化**（親より狭い入力しか受けない）、または `throw UnsupportedOperationException()` / 空実装で契約を破っていないか。
- インターフェース実装で、あるメソッドだけ `TODO()` や例外 throw で「実装しないことにしている」ものは履歴/事後条件違反の疑い。

### ISP
- **巨大 Repository インターフェース** を画面/UseCase ごとに一部メソッドだけ使い分けていないか（1つの `XxxRepository` に20メソッドあり各クライアントは2〜3個しか使わない）。→ ロール別インターフェースへの分割が処方。
- Hilt の `@Binds` で太い interface を注入し、利用側が大半を無視しているパターン。

### DIP
- **domain に interface・data に実装**、Hilt の `@Binds`/`@Provides` で bind、という基本形が守られているか。
- **上位層が下位層の実装クラスを import** していないか（`import tv.abema.data.xxx.XxxRepositoryImpl` が usecase/domain に現れる、など）。これは明確なDIP違反の兆候。
- コンストラクタ引数の型が `XxxRepository`（抽象）ではなく `XxxRepositoryImpl`（具象）になっていないか。

## verdict 判定フロー

原則ごとに上から適用する:

1. **その粒度でその原則が適用不能か？**（例: 関数のISP/LSP、モジュールのLSP）
   → Yes: **評価対象外**（理由を明記）。No: 次へ。
2. **原則から外れているか？**（上記チェック項目に該当する兆候があるか）
   → No: **遵守**。Yes: 次へ。
3. **具体的な変更シナリオで実害が説明できるか？**
   （「◯◯という変更が来たとき、この構造のせいで△△箇所を触る/バグる/理解困難になる」が具体的に言えるか）
   → **Yes: 違反**。実害の深刻さで重大度を付す（下記）。
   → **No（原則からは外れるが実害シナリオが弱い/限定的）: 軽微な逸脱**。

### 重大度（違反にのみ付す）

| 重大度 | 目安 |
|---|---|
| **high** | 頻繁に来る変更軸に直撃する / 実害が広範囲（多数の呼び出し元・複数モジュールに波及）/ バグ・回帰の温床になっている。 |
| **medium** | 実害シナリオは明確だが、変更頻度が中程度、または影響範囲が限定的。 |
| **low** | 実害は説明できるが、変更頻度が低い / 局所的で回避も容易。 |

### 保守的判定の原則（`solid-principles.md` 批判・限界セクションと整合）

- 実害シナリオが **投機的**（「将来〜かもしれない」）なだけなら **違反にしない**（軽微な逸脱以下）。
- **深いモジュール**（単純な公開面 + 複雑な実装）を、原則を理由に割って表面積を増やす提案はしない。分割の処方は、現に混在するアクター/変更軸が観測できる場合に限る。
- 1箇所しかない分岐、単一クライアントしかない太いインターフェースなど、**分離の受益者が現状いない** ケースは遵守または軽微な逸脱に留める。
