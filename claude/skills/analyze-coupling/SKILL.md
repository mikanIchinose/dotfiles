# Coupling Balance Analyzer（汎用）

任意のリポジトリでコンポーネント間の結合を均衡結合モデルで評価し、リファクタリング優先度を付ける。
理論の数値モデルは `references/theory.md`、統合強度の判定基準は `references/strength-criteria.md`、処方の選び方は `references/rebalancing.md` を正とする。

> **チーム編成**: 司令塔はこの skill を実行するメインエージェント。統合強度を精査するワーカーは **Opus**（`Agent` ツールで `model: opus` 指定）を**常に並列起動**する。

## 引数（すべて省略可）

- **粒度**: `module`（ビルドシステムのモジュール/ワークスペース） / `package`（言語のパッケージ・名前空間） / `dir`（ディレクトリ） / `service`（マイクロサービス等のデプロイ単位）。省略時はステップ0で自動推定。
- **N**: LLM 精査するワーストエッジ数。省略時は規模連動: `clamp(全エッジ数 × 0.3, 4, 30)`。
- **出力先**: レポートの出力パス。省略時は `tmp/docs/coupling-analysis-report.md`（ディレクトリがなければ作成）。

## ワークフロー

### ステップ0: 粒度の決定

リポジトリ構造を調べ、評価単位（書籍の「コンポーネント」）を推定する:

1. ビルドシステムのマルチモジュール構成（`settings.gradle(.kts)`、pnpm/npm/yarn workspaces、Cargo workspace、Go modules、Maven modules 等）があれば → `module`
2. モノリポでサービスディレクトリが並ぶ構成（`services/`, `apps/` + 個別デプロイ設定）→ `service`
3. 単一モジュールなら → 言語のトップレベル `package` または `src/` 直下の `dir`

推定結果（粒度と、それで数えたコンポーネント数）をユーザーに1行で報告してから進む。コンポーネント数が 3 未満または 200 超なら粒度が不適切な可能性が高いので、AskUserQuestion で確認する。

### ステップ1: 依存グラフの構築

**ビルドツールの依存グラフ出力を優先**し、なければ import 解析にフォールバックする:

| 環境 | 依存グラフの取得例 |
|---|---|
| Gradle | `./gradlew :mod:dependencies --configuration api/implementation`、または `settings.gradle` + 各 `build.gradle` の `project(...)` 宣言を grep |
| npm/pnpm workspaces | 各 `package.json` の workspace 依存 |
| Go | `go mod graph`（モジュール間） / `go list -deps`（パッケージ間） |
| Rust | `cargo tree` / `cargo metadata` |
| Maven | 各 `pom.xml` の `<dependencies>` |
| フォールバック | 各コンポーネント内ソースの import/require/use/include 文を grep し、他コンポーネントのシンボルへの参照をエッジ化 |

各エッジについて **共有シンボル情報** も収集する（後の強度判定の材料）: 下流が import している上流のシンボル数・種別（データ型/インターフェイス/具象クラス/関数）、`internal`/`impl`/非公開パッケージへの参照疑い。

各コンポーネントについて: ファイル数、おおよその LOC、直近12ヶ月の git コミット数（`git log --since="12 months ago" --format=%H -- <path>` の件数）。

### ステップ2: 距離マッピングの定義

距離は「最も近い共通祖先が浅いほど遠い」原則（書籍8.1.3）で、**このリポジトリ専用のマッピング表**を作る。書籍10.3の例示スケール（同一クラス=1〜別ベンダー=10）を粒度に合わせて再配分し、距離が定数化しないようにする。目安:

- 同一親ディレクトリ/同一機能グループ内 = 3
- 共通基盤・ユーティリティコンポーネントとの結合 = 4
- 隣接アーキテクチャレイヤー間 = 5
- 非隣接レイヤー間 = 7
- レガシー領域・別リポジトリ・別デプロイ単位をまたぐ = 8〜9

作ったマッピング表はレポートに必ず記載する（スコアの再現性のため）。チーム境界（CODEOWNERS 等）が明確なら、別チーム所有をまたぐエッジは +1 してよい（根拠をレポートに書く）。

### ステップ3: 変動性の評価（2系統）

各コンポーネントについて:

1. **git 実績**: 直近12ヶ月コミット数のパーセンタイル順位を 1-10 に写像。
2. **ドメイン分析**: コンポーネントの役割からサブドメイン種別を判定 → 基礎変動性（コア=10 / 支援・汎用=3 / レガシー=1）。
3. **補正**: 偽陽性（fix ばかりで仕様は安定 → 下げる）と偽陰性（コアなのに低コミット → 「変更困難ゆえ凍結」の疑い。フラグを立て、優先度はむしろ上げる）をチェック。

詳細は `references/theory.md` の変動性の節。

### ステップ4: 暫定ランキングとワーストN抽出

各エッジに、ステップ2のマッピング表を適用した距離 D と、ステップ3で算出した上流コンポーネントの基礎変動性を暫定 V として割り当てる（実効変動性への差し替えはステップ6で行う）。
次に各エッジに暫定強度を置く（ヒューリスティック: internal/非公開参照疑いあり→8-10、具象クラス・振る舞いの共有→6-8、データ型のみ→3、interface+専用DTOのみ→1-2）。
暫定均衡度 `max(|S-D|, 10-V) + 1` の昇順（同点はメンテナンス労力 `S×D×V` 降順）でソートし、上位 N エッジを精査対象にする。**精査対象外のエッジ数をレポートに明記する。**

### ステップ5: 統合強度を Opus ワーカーで並列精査

精査対象エッジをモジュールペア単位でグルーピングし、`Agent` ツール（`model: opus`）を **1メッセージ内で複数並列起動**する。ペア数が多い場合は数ペアずつ1ワーカーに割り当てる。

各ワーカーに渡すもの:
- `references/strength-criteria.md`（このスキルディレクトリ内。絶対パスで指示）
- 対象エッジの from/to コンポーネントのパスと、ステップ1で集めた共有シンボル情報
- 指示: 実ソース（下流の import と参照箇所、上流の宣言）を読み、判定フローチャートで **統合強度レベル + スコア(1-10) + コナーセンス種別 + 根拠（具体的なシンボル名/パス） + 処方の方向性** を確定して返す。

### ステップ6: 実効変動性の伝播と確定再計算

1. 上流の変動性を統合強度で減衰させて下流へ伝播（減衰係数: 侵入≈1.0 / 機能≈0.8 / モデル≈0.4 / コントラクト≈0.1）。エッジの変動性 V には上流の実効変動性を使う。
2. 確定した S / D / V で全指標を再計算: 均衡度（主指標）、モジュール性、メンテナンス労力、安定性・変更コスト・大域的/局所的複雑性（2値、閾値は S≧6 / D≧6 / V≧6）。式は `references/theory.md`。

### ステップ7: レポート出力

出力先（既定 `tmp/docs/coupling-analysis-report.md`）に日本語で出力。構成は `references/report-template.md` に従う:
- エグゼクティブサマリ + 最優先トップ10（各エッジに処方: **近づける / 強度を下げる / 分離する / 許容(監視のみ)** のどれを、なぜ、具体策。変動性は方向選択の判断材料。選び方は `references/rebalancing.md` の判断フロー）
- エッジ別詳細（確定スコア + 根拠）
- コンポーネント別サマリ（fan-in/fan-out、実効変動性、平均均衡度、偽陰性フラグ）
- 距離マッピング表と精査対象外エッジ数（打ち切りの可視化）

チャットには要約（トップ3と全体所見）のみを返す。

- `references/theory.md` — 3次元の定義・数値スケール・全評価式・実効変動性の伝播・フラクタル性（粒度非依存の根拠）
- `references/strength-criteria.md` — 統合強度の4レベル判定フローチャート・コナーセンスによる度合い調整（ワーカーに渡す）
