# 統合強度 判定基準（ワーカー用）

出典: Vlad Khononov『ソフトウェア設計の結合バランス』第7章の統合強度モデルを、本スキル用の判定手順として再構成したもの。
このファイルは精査ワーカーが1つのコンポーネントペアの統合強度を確定させるための自己完結した基準。
実ソースを読み、下記フローチャートで **4レベル + スコア(1-10) + コナーセンス種別 + 根拠** を確定させること。

---

## 統合強度とは

- 結合された2コンポーネント間で共有される「知識」の質。**強いほど、片方の変更が他方に波及しやすい**（変更理由の共有）。
- 構造化設計のモジュール結合とコナーセンスを1つに統合したモデル。
- 評価は2軸:
  1. **インターフェイスの種類** = レベル（どの結合か）
  2. **インターフェイスの複雑さ** = 度合い（レベル内の微調整。コナーセンス由来）
- ここで言う「インターフェイス」は統合の方法のこと。言語の `interface` 構文とは別概念で、物理的接続がなくても「一緒に変更される」関係はインターフェイスとみなす。

## 4レベルとスコア

| レベル | スコア | 共有する知識 |
|---|---|---|
| 侵入結合 (Intrusive) | **10** | 上流の実装詳細（最強・最悪） |
| 機能結合 (Functional) | **8**（対称は **9**） | ビジネスルール・ロジック・振る舞い |
| モデル結合 (Model) | **3** | ビジネスドメインのデータモデル（構造のみ） |
| コントラクト結合 (Contract) | **1** | 統合専用モデル（最弱・最良） |

スコアは度合いでレベル内を微調整する（例: モデル結合でも位置のコナーセンスが強ければ 3→4 寄り、機能結合が実行順序依存のみなら 8 の下限）。

---

## 判定フローチャート（上から評価、最初に該当したレベルを採用 = 強い方優先）

```
Q1. 下流が上流の「統合用に意図されていない実装詳細」に依存しているか?
    - 非公開・internal・impl 相当のシンボルへのアクセス（言語の可視性を回避する手段を含む:
      リフレクション、friend/@VisibleForTesting の乱用、アクセス制限の抑制アノテーション）
    - 上流コンポーネントが所有するDB/永続化ストア/設定ファイルへ、統合APIを経由せず直接read/write
    - 上流の非公開コードのコピー&ペースト利用
    └ YES → 【侵入結合 = 10】
    └ NO ↓

Q2. 2コンポーネントが「機能・ビジネスロジック・振る舞い・実行順序・トランザクション」を共有しているか?
    (データ構造だけでなくロジックまで)
    - 同じビジネスルールを両者が重複実装している → YES
    - 呼び出しの実行順序/タイミングに暗黙の依存がある → YES
    - 全成功か全失敗のトランザクション境界を共有 → YES
    - 同じデータを同じ整合性ルールで読み書き → YES
    └ YES → 【機能結合 = 8】
              ├ 同じ機能を両者が独立実装(DRY違反)なら【対称機能結合 = 9】
              └ 度合い(実行<タイミング<トランザクション<値<同一性)で 8 前後に微調整
    └ NO ↓

Q3. 2コンポーネントが「ビジネスドメインのデータモデル(データ型・エンティティ)」を共有しているか?
    (振る舞いは共有せず、型・プロパティ構造だけ。上流の内部モデルが公開APIに露出し下流が依存)
    └ YES → 【モデル結合 = 3】
              └ 度合い(名前<型<意味<アルゴリズム<位置)で 3 前後に微調整
    └ NO ↓

Q4. 2コンポーネントが「統合専用に整形されたモデル/コントラクトのみ」で通信し、内部実装モデルは
    カプセル化されているか? (interface + 専用DTO、Command/Query API、proto/OpenAPI等の標準プロトコル)
    └ YES → 【コントラクト結合 = 1】
              └ ただしDTOが内部モデルの単なる鏡写し(same shape)なら モデル結合(3) へ格上げ
    └ NO → 統合が薄い/情報不足。判定できなければその旨を根拠に記す
```

## 分かれ目の要点

- **侵入 vs モデル**: 同じ「DB読み取り」でも、統合として**意図・許可**されていればモデル結合、勝手なアクセスなら侵入結合。ORMのような意図されたリフレクション利用は侵入結合ではない。
- **機能 vs モデル**: データ構造の共有はモデル結合、**振る舞い/ルールの共有は機能結合**。
- **モデル vs コントラクト**: 統合モデルが内部実装知識をカプセル化して隠しているか（深いか）。鏡写しDTOはモデル結合側。
- **強い側を優先**: 複数該当なら 侵入 > 機能 > モデル > コントラクト。

## 度合い（コナーセンス種別）— スコア微調整の材料

### 機能結合の度合い（弱→強、動的コナーセンス）
1. 実行のコナーセンス（実行順序依存）→ 8 の下限
2. タイミングのコナーセンス（順序 + タイミング/期間制約）
3. トランザクションのコナーセンス（全成功か全失敗）
4. 値のコナーセンス（値が一緒に等しく変わる）
5. 同一性のコナーセンス（同じデータセットで強整合）→ 8 の高め
6. 対称機能結合（同じ機能を独立実装、DRY違反）→ **9**

### モデル結合・コントラクト結合の度合い（弱→強、静的コナーセンス）
名前 < 型 < 意味 < アルゴリズム < 位置（位置が最も暗黙的で統合ミスしやすい）
- コントラクト結合の**最強の度合いでも、モデル結合の最弱より共有知識が少ない**（→ コントラクトは 1 に据え置ける根拠）。

### コントラクト結合の「深さ」判定（モデルとの境界）
- コンポーネントを長方形（面積=実装詳細、底辺=公開インターフェイス）とみなす。深い（縦長）＝知識を隠している＝良いコントラクト。浅い＝隠せていない。
- DTO/専用モデルが内部モデルと同形（same shape）なら、名前/型のコナーセンスはあっても知識をカプセル化しておらず、実質モデル結合(3)。

---

## 非同期実行の注意

- **同期/非同期は統合強度に影響しない**。何を（どんな知識/モデルを）やり取りするかで判定する。
- メッセージキュー/イベントバス/リアクティブストリーム経由でも同期呼び出しと同じ基準で分類。「非同期だから疎結合」と早合点しない。
  - キュー経由でも統合専用モデルならコントラクト(1)、ドメインモデル形式ならモデル(3)、タイミング待ちなら機能(8)、バスが内部実装詳細なら侵入(10)。

---

## 構造化設計との対応（逆引き）

- 侵入結合 = 内容結合(content)
- 機能結合 = 共通(common)・外部(external)・制御(control)結合
- モデル結合 = スタンプ結合(stamp)
- コントラクト結合 = データ結合(data)

---

## ワーカーへの指示（成果物フォーマット）

各コンポーネントペアについて、実ソース（下流の import 文と参照箇所、上流の宣言）を読んで以下を確定して返すこと:

- **統合強度レベル**: 侵入 / 機能 / 対称機能 / モデル / コントラクト
- **スコア(1-10)**: 上表 + 度合いで微調整した確定値
- **コナーセンス種別**: 上記の度合い分類から該当するもの
- **根拠**: どの import / どのシンボル / どの参照パターンからそう判定したか。具体的なシンボル名やファイルパスを引用
- **処方の方向性**: 強度を下げる余地があるか（例: 内部モデル露出→DTO化でコントラクト化、重複ルール→共通コンポーネント抽出）
