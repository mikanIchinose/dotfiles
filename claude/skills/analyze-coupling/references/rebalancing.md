# 処方の選び方 — 再均衡化のパターン集

出典: Vlad Khononov『ソフトウェア設計の結合バランス』第11章（結合の再均衡化）・第12章（成長のダイナミクス）・第13章（実践ケーススタディ）を本スキル用に再構成したもの。
レポートの「処方」欄を書くとき、このファイルの判断フローとパターンに従う。

---

## 大原則

**強度と距離は反比例させ、変動性で許容度を調整する。**

- 強く結合するものは近くに置く。遠くに置くものは弱く結合する（強度 XOR 距離）。
- 変動性が低ければ、多少の高強度・遠距離は許容される（OR NOT 変動性）。
- 最悪の組み合わせは「高強度 × 遠距離 × 高変動」の三重苦。連鎖的変更コストが最大化する。**これを検出したら最優先の是正対象。**

### 是正の方向は本質的に3つ

1. **強く結合したものが遠い** → **近づける**（移設・統合・機能軸への再編・集約化）
2. **遠距離を跨ぐ結合** → **強度を下げる**（統合コントラクト/DTO化、ポート&アダプター、公開イベントの最小化）
3. **知識を共有しないものが近くに同居** → **離す**（SRP/ISP分離、機能軸パッケージング、メソッド抽出）

変動性そのものは設計では下げられない（ビジネスの性質で決まる）。高変動には距離か強度で応じる。

---

## 判断フロー

1. **変動性を最初に見る（羅針盤）**: 上流が低変動（支援/汎用/レガシー）なら、理論上の「臭い」があっても **「許容 / 監視のみ」が正解になりうる**。過剰なリファクタ提案を抑える。同じ強度・距離の2エッジでも変動性で優先度が変わる。
2. **診断の型で分類**:
   - **大域的複雑性**（高強度 × 遠距離）→ 処方1（近づける）か処方2（強度を下げる）。どちらを選ぶかは「その2つは本当に一緒に進化すべきものか」で決める: 一緒に進化すべき（同じ機能・同じトランザクション）なら近づける。独立に進化すべきなら強度を下げる。
   - **不安定**（高変動 × 高強度）→ 処方2（強度を下げる: コントラクト化・腐敗防止層）か、一緒に進化すべきなら処方1（距離を消す: 集約・移設）。
   - **局所的複雑性**（低強度 × 近距離）→ 処方3（分離・整理）。
3. **複雑性の付け替えに注意**: 距離の調整だけでは、局所的複雑性を大域的複雑性に（またはその逆に）**変換するだけ**で終わることがある。処方を書く前に「これは複雑性を本当に減らすか、場所を移すだけか」を問い、付け替えにしかならない場合はその旨を警告として書く。根本解は新しい抽象化レベルの導入（後述）。

## 変更要因別のパターン（第11章）

検出した悪い結合が「何によって生じたか」を推定すると処方の精度が上がる。

| 変化の種類 | 動いた次元 | 処方 |
|---|---|---|
| 機能拡張で統合が増えた | 強度↑ | 強度を元に戻す（リファクタ）か距離を縮める。低変動なら放置も可 |
| サブドメイン種別の移行（支援/汎用→コア） | 変動性↑ | 距離を消す（重複の集約・移設）か強度を下げる（専用API化: 侵入→コントラクト） |
| サービス分割・組織変更 | 距離↑ | 境界を越えて共有する知識を減らし、モデル結合→コントラクト結合へ下げる |

- **戦術的変更**（バグ修正・実装改善 = How の変更）は通常再均衡化不要。**戦略的変更**（問題の再定義・新機能・事業/組織/環境の変化 = What の変更）が結合次元を乱す。
- **氷山の警告**: 「一見無害な if 文の追加」がドメインモデルの重要な変更を隠していることがある。無害に見える変更ほど、どの次元を乱していないか疑う。
- **強度不足**もある: 期待される一貫性（例: 即時反映・トランザクション整合）を弱い結合が満たせていない場合は、機能を必要とする側へ**移設して距離を縮め**、必要な強度を確保する。
- **意図的な重複は選択肢**: 単純で変わらないロジックなら、共通化せず重複させたまま放置してよい（低変動が高強度×遠距離を相殺）。DRY の機械的適用より均衡を優先。

## ケーススタディ由来の診断→処方パターン（第13章）

| 症状（検出パターン） | 診断 | 処方 |
|---|---|---|
| 内部モデル（エンティティ/内部イベント）をそのまま別コンポーネント・別チームに公開 | モデル結合 × 遠距離。高変動なら最優先 | 統合専用コントラクト（DTO/公開イベント/API境界の型）を挟み内部モデルを隠蔽 |
| 同等の強度・距離だが上流が支援/汎用サブドメイン | バランスは取れている | **是正不要**。監視のみ |
| 1つの機能変更が技術レイヤー/複数モジュールに横断的に波及 | 機能結合が距離で引き離されている（技術軸分割のねじれ） | 機能軸（feature/ドメイン軸)で凝集させ直す（垂直スライス化）。同時に変わるものを近くに |
| ビジネスロジックが具体的なインフラ実装（DB・外部API・フレームワーク）を直接知っている | 機能結合 + インフラの変動性を偶発的に継承 | ポート&アダプター/DIP: インターフェイスを高位側に置き、実装をインフラ側へ押し出す |
| オブジェクトが直接参照で絡み合い、トランザクション/ロード境界が曖昧 | 過剰な強度（不要なトランザクション結合を許す設計） | 集約パターン: トランザクション整合が本当に必要な単位で境界を引き、境界外は ID 参照に置換 |
| パッケージ/ディレクトリが技術役割別（entities/, repositories/…）で機能変更が散らばる | 近距離 × 低強度の同居 + 機能結合の散在 | 機能別パッケージへ再編（スクリーミングアーキテクチャ） |
| 1クラス/1インターフェイスに、一緒に変わらないメソッドが同居 | SRP/ISP 違反 = 近距離 × 低強度 | 知識を共有しない操作を別インターフェイス/クラスに分離 |
| 1メソッドが無関係な複数の関心事を処理 | メソッド内の局所的複雑性 | メソッド抽出（最小粒度でも同じ原則） |

既知の設計原則はすべて均衡結合モデルの特殊ケース: レイヤード→VSA は強度×距離の再バランス、DIP/ポート&アダプターは遠距離結合のコントラクト化、集約はトランザクション結合の距離最小化、SRP/ISP は低強度同居の分離。

## 成長視点の優先度補正（第12章）

- 相互作用（結合）はコンポーネント数に対して**スーパーリニア**（n(n-1)/2 的）に増えるが、人間の認知能力は一定。ある規模で「巨大な泥団子」化する。
- したがって「今バランスが取れているか」だけでなく **「成長したときどこが最初に壊れるか」** を見る: fan-in/fan-out が大きく成長が続いているコンポーネント、相互作用密度の高いクラスタを優先的に指摘する。
- 強度・距離の微調整で複雑性の付け替えにしかならない段階では、**新しい抽象化レベルの導入**（包括的モジュールを設けて責任を再割り当てする構造変更）を処方として提案する。

## レポートの処方欄の書き方

各エッジについて:
1. 診断（大域的複雑性 / 不安定 / 局所的複雑性 / 三重苦）を明記
2. 3方向（近づける / 強度を下げる / 分離する）のどれを選ぶか、**なぜそれか**（一緒に進化すべきか否か、変動性の水準）
3. 具体策（上の表のパターンから。対象シンボル・移動先まで書く）
4. 「許容/監視のみ」の判断も処方として明記する（低変動が根拠なら根拠を書く）
5. 複雑性の付け替えにしかならない場合は警告を添える
