# 価値のあるドキュメントを書く

## このスキルが守る一点

**読者のペインを解決する。** これが最優先事項であり、他のすべての判断基準になる。

文章の良し悪しは、表現の巧みさではなく「狙った読者の困りごとが解決したか」で決まる。だから書き始める前に「誰の、どんな困りごとを、どう解いた状態にするか」を言語化する。ここが曖昧なまま書くと、情報は正しくても誰にも刺さらない文章になる。

このスキルは、ゼロから書くときも、既存の下書きを推敲・レビューするときも使える。下書きがある場合は「書く前に決める3つのこと」が言語化されているかをまず確認し、抜けていれば著者に問い直すところから始める。

## 書く前に決める3つのこと

本文を書く前に、次の3つを文章で書き出す。箇条書きの単語ではなく、文で書く。文にできない時点で、まだ解像度が足りていない。

### 1. 読者ペルソナ

できる限り具体的に1人に絞って言語化する。「初心者全般」のような広いペルソナは、結局誰にも届かない。

書き出す項目の例:
- 職種・経験レベル（例: Android アプリを2年書いているが、KMP は触ったことがない）
- いま直面している状況（例: 既存アプリのロジックを iOS と共有したいと上司に言われた）
- 前提知識（何を知っていて、何を知らないか）
- どんな検索語でこの記事にたどり着いたか

検索語まで想像すると、読者の困りごとが具体的な言葉になる。

### 2. 読者のペイン

ペルソナが「いま」抱えている困りごとを、読者自身の言葉で書く。著者視点の「〜を解説する」ではなく、読者視点の「〜がわからなくて手が止まっている」で書く。

- ✗ KMP のセットアップ方法を解説する
- ✓ 公式ドキュメント通りにやってもビルドが通らず、どこで間違えたのか切り分けられない

### 3. 解決された状態

読み終えたとき、読者が何をできるようになっているか／どんな状態になっているかを書く。これが記事のゴールであり、**本文の取捨選択の基準**になる。ゴールに寄与しない情報は、正しくても削る。

- 例: 自分のプロジェクトで KMP モジュールをビルドできる状態になり、つまずいたときにどこを見ればいいか判断できる

## 文章構造を選ぶ

読みやすさに銀の弾丸はない。ペインの種類によって最適な型が変わる。下の早見表で当たりをつけ、`references/structures.md` で選んだ型の骨格と書き方を確認してから本文に入る。

| 読者のペイン | 向いている型 |
|---|---|
| エラー・不具合を解決したい | 問題解決型 |
| 手順をなぞって動くものを作りたい | チュートリアル型 |
| 選択肢があって決められない | 比較・選定型 |
| 概念・仕組みを理解したい | 解説型 |
| 誰かの実体験から学びたい | 体験記・ふりかえり型 |

迷ったら、複数の型を組み合わせてもよい（例: 解説型の導入＋チュートリアル型の本編）。型は読者を導くための道具であって、埋めるべきテンプレートではない。

## 価値の源泉 — 著者の魂を載せる

**公開情報をただまとめただけの文章に価値はない。** 公式ドキュメントの劣化コピーなら、読者は一次情報を読めばいい。

価値が生まれるのは、まとめ方に著者独自の視点が入っているときだけだ。視点・経験・感想 ——「著者の魂」—— こそが、その文章をその人にしか書けないものにする。

書きながら、次を意識して足す:
- **なぜ調べたか**: どんな文脈でこの問題に出会ったか
- **どこでハマったか**: 公式通りにやって詰まった点、ドキュメントに書いていなかった落とし穴
- **試した結果**: 自分の環境で実際にどうなったか（数字・ログ・スクショ）
- **既存情報との差分**: 他の記事や公式と違う、自分が見つけた事実や近道
- **どう感じ、次にどうするか**: 評価と、その先の判断

これらが1つも無い下書きを見つけたら、著者に問う:「この記事の中で、あなたにしか書けない部分はどこですか？」

## 推敲チェックリスト

書き終えた後、または下書きをレビューするとき、次を1つずつ確認する。引っかかったら直す。

### 価値
- [ ] 著者独自の視点・経験・感想が入っているか。公開情報のまとめで終わっていないか
- [ ] 「解決された状態」に到達できる内容になっているか。ゴールに寄与しない記述を削ったか

### 事実と感想を分ける
読者は事実をもとに自分で判断したい。事実に感想が混ざると、どこまで信用していいか分からなくなる。
- [ ] 事実（観測・計測・仕様）と感想（評価・印象）が混ざっていないか
- 分け方の例:
  - 事実: ベンチマークでは A が B より 1.8 倍速かった。
  - 感想: この差は体感でもはっきり分かり、移行する価値があると感じた。
- 文・段落を分けるか、「— ここからは感想だが」のように明示する。

### 体言止め（名詞止め）を使わない
- [ ] 文末を名詞・体言で終えていないか
- 体言止めは一見リズムが出るが、述語が省略されるぶん主張がぼやける。それが事実なのか、願望なのか、感想なのかが消えてしまう。述語まで書き切ることで、書き手は自分が何を言いたいのかを確定させられる。
  - ✗ 重要なのはキャッシュの設計。
  - ✓（事実として）キャッシュ設計が応答速度を左右する。
  - ✓（主張として）重要なのはキャッシュの設計だと考えている。

### あいまい言葉を使わない
- [ ] 考えを表現・組み立てる場面で、あいまい言葉に逃げていないか
- あいまい言葉は、書き手の理解の低さをそのまま読者に渡してしまう。「適切に設定する」と書くとき、たいてい書き手自身も何が適切か言語化できていない。具体に置き換える過程で自分の理解が深まり、読者も再現できるようになる。

| あいまい言葉 | 置き換え方 |
|---|---|
| いろいろ／さまざま | 具体的に列挙する（3つなら3つ書く） |
| 適切に／ちゃんと／しっかり | 何を、どの基準で、どう操作するかを書く |
| たぶん／〜かもしれない（事実の場面） | 確かめて言い切る。確かめられないなら「未確認だが」と明示する |
| 便利／いい感じ／すごい | 何がどう良いのか、比較対象との差分を書く |
| 〜と思います（事実の場面） | 事実は言い切る。感想なら感想として分けて書く |

### 読みやすさ
- [ ] 選んだ型の流れに沿っているか。読者が迷子になる飛躍がないか
- [ ] コード・コマンド・ログは、読者が自分の環境で再現できる粒度で書かれているか

## 進め方

新規執筆なら、まず「書く前に決める3つのこと」を著者と一緒に言語化し、合意してから本文に入る。ここを飛ばさない。

推敲・レビューなら、上のチェックリストで引っかかった箇所を、**なぜ直すべきか**を添えて具体的に指摘する。「ここを直して」だけでなく「体言止めで主張がぼやけているので、述語まで書くとこうなる」と直し方まで示す。
