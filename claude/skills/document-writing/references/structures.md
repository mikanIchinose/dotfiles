# 文章構造テンプレート集

読者のペインに合わせて型を選ぶ。型は読者を導く道具であり、機械的に埋める枠ではない。各見出しは目安なので、不要なら削り、必要なら足す。骨格を守りつつ、各セクションに「著者の魂」（経験・つまずき・感想）を載せること。

---

## 問題解決型

**向くペイン**: エラー・不具合・性能問題を解決したい。

```
## 何が起きていたか（症状）
読者が検索する症状を、再現条件つきで具体的に。エラーメッセージは原文のまま。

## 原因
なぜそれが起きるのか。仕組みのレベルで説明する。ここが分かると応用が利く。

## 解決策
最小の手順で。コードは差分が分かる形で。

## なぜこれで直るのか
原因と解決策をつなぐ。対症療法で終わらせない。

## ハマったこと・補足（著者の魂）
公式に載っていなかった落とし穴、自分の環境固有の事情、別の解き方との比較。
```

書き方のコツ: 「症状 → 原因 → 解決」の順を崩さない。先に解決策だけ書くと、読者は応用が利かず、状況が少し違うだけで詰まる。

## チュートリアル型

**向くペイン**: 手順をなぞって、動くものを自分で作りたい。

```
## このチュートリアルのゴール
完成したとき何ができるか。スクショや完成物のリンク。

## 前提
必要な環境・バージョン・前提知識。ここを明示すると「自分でもできそう」か判断できる。

## 手順
1ステップ＝1つの操作。各ステップの最後に「ここで何が起きたか」を確認する記述を入れる。
コマンドとその出力をセットで示す。

## 動作確認
読者が「正しくできた」と自分で確認する方法。

## つまずきやすいポイント（著者の魂）
自分が実際に詰まった箇所と抜け方。バージョン差で変わる点。
```

書き方のコツ: 手順の粒度を一定に保つ。読者は自分の環境で1行ずつ実行する。コピペで動く粒度を死守し、省略するなら省略したと明示する。

## 比較・選定型

**向くペイン**: 選択肢が複数あって、どれを選べばいいか決められない。

```
## 何を選ぼうとしているのか
読者が置かれた状況。選定が必要になった背景。

## 評価軸
何を基準に比べるか。軸を先に出すと、読者は自分の優先順位に当てはめられる。

## 比較
軸ごとに各選択肢を評価。表が有効。事実（仕様・計測）と感想（使い心地）は分けて書く。

## 結論と、その理由
著者ならこれを選ぶ、という立場を理由つきで。読者の状況が違えば結論も変わる点を添える。

## 実際に使ってみた所感（著者の魂）
カタログスペックでは分からない、使って初めて分かったこと。
```

書き方のコツ: 「どれも一長一短です」で逃げない。読者は決められなくて困っている。著者の立場を理由とともに示すことが価値になる。前提が違えば結論が変わる条件もあわせて書く。

---

## 解説型

**向くペイン**: 概念・仕組みを理解したい。

```
## 全体像
まず俯瞰。これから説明するものが、何の役に立ち、どこに位置するか。

## 構成要素
全体を分解して1つずつ。それぞれが何のために存在するか（why）を添える。

## 具体例
抽象的な説明を、動くコードや身近なたとえで地に足をつける。

## よくある誤解・つまずき（著者の魂）
自分が誤解していた点、理解が腑に落ちた瞬間のきっかけ。
```

書き方のコツ: 抽象→具体の順で、抽象だけで終わらせない。読者が「で、結局どう使うの」とならないよう、必ず具体例で着地させる。

---

## 体験記・ふりかえり型

**向くペイン**: 誰かの実体験から、判断材料や勇気をもらいたい。

```
## 状況
何に取り組んでいたか、なぜそれをやったか。読者が自分と重ねられる背景。

## やったこと
時系列で。判断の分岐点では「なぜそう決めたか」を書く。

## 起きたこと
結果を事実として。うまくいったこと、いかなかったこと両方。数字があれば数字で。

## 学び（著者の魂）
次に同じ状況に立ったらどうするか。読者が持ち帰れる教訓。
```

書き方のコツ: この型は「著者の魂」そのものが主役になる。事実（起きたこと）と学び（解釈）を混ぜず、読者が事実から自分なりの学びを引き出せる余地を残す。
